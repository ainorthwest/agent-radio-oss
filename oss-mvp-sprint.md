## What this sprint is

This sprint creates a **new public repo: `ainorthwest/agent-radio-oss`**.

It is not a refactor of the existing `ainorthwest/agent-radio` repo. The existing repo continues to evolve as the proprietary Lightcone Studios production stack (with its MLX engines, Steward agent, AINW-flavored shows, and the Orpheus/Dia/CSM/Qwen3 voice cast). The OSS repo is the **clean, hardware-agnostic, commercially-deployable distribution** that anyone in the world can clone and use to run a real internet radio station.

The goal is not to ship a toy. The goal is to ship something where **any operator with any hardware can stand up a commercially-viable autonomous radio station out of the box.** That commercial-deployability constraint shapes every model and library choice below.

Why a fresh repo and not a fork:
1. **Clean license boundary.** OSS repo only contains commercially-clean code, weights, and assets. No CC-BY-NC liabilities, no Steward IP, no AINW-specific casts.
2. **Clear identity.** "agent-radio-oss" tells contributors what they're getting. "agent-radio" is the production stack; "agent-radio-oss" is the platform.
3. **No accidental contamination.** Once the repos diverge, work in one doesn't bleed into the other. Code can be ported deliberately, not osmotically.
4. **The bifurcation is real from day one.**

The existing `agent-radio` repo gets a `BIFURCATION.md` pointing at `agent-radio-oss` and explaining the relationship. That's the only change to the production repo this sprint.

## Model & library shortlist (decisions locked)

| Capability | Library | License | Hardware story | Why this pick |
|---|---|---|---|---|
| **TTS** | Kokoro ONNX | Apache 2.0 (code + weights) | CPU / CUDA / ROCm / CoreML via ONNX Runtime providers | 82M params, runs anywhere, named voices (no PII), production-grade quality |
| **STT** | whisper.cpp | MIT | CPU / CUDA / ROCm via Vulkan / Metal / CoreML — all backend flags explicit | Most hardware-portable Whisper, no Python dep tangle, binary backend exposes hardware lessons |
| **Music** | Stable Audio Open via `stable-audio-tools` | **Stability AI Community License (commercial-permitted up to revenue thresholds)** | PyTorch, CUDA / ROCm / MPS / CPU | Commercial-clean (vs MusicGen's CC-BY-NC), text-to-music, ~1B params |

**Why these three together:** they teach the same hardware-backend lesson three different ways.
- ONNX Runtime exposes providers as explicit strings (`["CUDAExecutionProvider", "ROCMExecutionProvider", "CoreMLExecutionProvider", "CPUExecutionProvider"]`)
- whisper.cpp exposes backends as compile flags / runtime args (`-DGGML_VULKAN=ON`, `-DGGML_HIPBLAS=ON`, `-DGGML_CUDA=ON`)
- PyTorch (Stable Audio Open) exposes them via `device = "cuda" / "mps" / "cpu"` with HIP-as-CUDA aliasing for ROCm

A contributor who learns to run all three on AMD has learned three different abstractions over the same GPU. That's the educational point.

**Decisions locked, no more debate this sprint:**
- whisper.cpp ✓
- Kokoro ONNX ✓
- Stable Audio Open ✓ (replaces MusicGen because OSS goal is commercial deployability)
- AMD-first via Hinoki RX 9070 ✓
- NVIDIA deferred to long-plan ✓
- MusicGen NOT in OSS MVP — its CC-BY-NC weights would foreclose commercial use

## North star for the week

**By end of sprint, a stranger with an AMD GPU + Ubuntu can `git clone github.com/ainorthwest/agent-radio-oss`, follow the README, and run their own commercial autonomous internet radio station — three Kokoro hosts reading AI news, with Stable Audio Open music beds, transcripts auto-generated by whisper.cpp, evaluated for intelligibility, packaged into an MP3 + RSS feed.**

"Haystack News" is the demo show because:
- AI news digest is universally legible — anyone gets the format
- The show is editorial format, not voice-locked. OSS uses Kokoro voices; production uses Orpheus/Dia
- Demonstrates the full creative loop: feed → curator → script → render → music → quality assessment → MP3 + transcript + RSS
- "AI talks about AI on AMD hardware" is a stronger launch story than "lofi beats"

## What success looks like

```bash
git clone git@github.com:ainorthwest/agent-radio-oss.git
cd agent-radio-oss
bash scripts/setup-amd.sh         # or setup-cpu.sh, setup-mac.sh, setup-cuda.sh

# Edit config/feeds.yaml with your news sources
# Add your OpenRouter key to .env (or use canned sample script)

uv run radio render --show haystack-news
# → output/episodes/{date}/episode.mp3
# → output/episodes/{date}/transcript.txt
# → output/episodes/{date}/transcript.srt
# → output/episodes/{date}/quality.json (LUFS, MOS, WER, etc.)

uv run radio publish --target r2          # optional
uv run radio publish --target podcast     # optional — generates RSS
```

…and the operator has a working autonomous AI radio station. Generated on their hardware. Commercially deployable.

For the OSS launch we need:
1. **One verified cross-platform render path on AMD ROCm** (Hinoki = primary dev environment)
2. **Verified CPU fallback** (any Linux, Mac, WSL works without GPU)
3. **Verified Apple Silicon path** (Shiro, via CoreML for Kokoro and Metal for whisper.cpp)
4. **A README that doesn't lie** about what works where
5. **Smoke tests anyone can run** to validate their install
6. **Per-hardware bring-up docs** capturing the quirks of each environment

### Day 2 (Tue) — Hinoki ROCm bring-up + Kokoro on every backend

**Goal:** Kokoro renders on Hinoki via every applicable ONNX provider, and on Shiro via CoreML, and on CPU.

The 9070 + ROCm path is expected to work — RDNA4 has official ROCm 7.0.2 support — but the stack is recent enough that we should budget for some massaging. Driver install, kernel module versions, and the `HSA_OVERRIDE_GFX_VERSION=12.0.0` flag are the usual culprits. **Documenting the massaging IS the deliverable** — those docs are what makes the OSS repo usable for the next AMD operator.

**Tasks:**
- SSH to Hinoki: `ssh hinoki`
- Install ROCm 7.0.2: `sudo apt install rocm-dev rocminfo` per AMD docs, `rocminfo | grep gfx` should show `gfx1201`
- `git clone agent-radio-oss` and `uv sync --extra tts --extra quality`
- Render a Haystack News audition (one 30-second segment) with Kokoro:
  - On CPU: `KOKORO_PROVIDER=CPUExecutionProvider`
  - On ROCm: `KOKORO_PROVIDER=ROCMExecutionProvider` (or `MIGraphXExecutionProvider` if ROCm provider is finicky on RDNA4)
  - Verify GPU engagement via `rocm-smi` showing utilization during render
- On Shiro: render same audition with `CoreMLExecutionProvider`
- On a clean Ubuntu Docker container: render with `CPUExecutionProvider` to verify zero-GPU baseline
- Run `uv run python -m src.quality output/audition.wav` on each — confirm scores within 0.1 of each other (same model, same input, different hardware = same output)
- Capture every install step, every quirk, every workaround in `docs/hardware/amd-rocm.md`, `docs/hardware/apple-silicon.md`, `docs/hardware/cpu.md`

**Exit criteria:** Kokoro WAV produced from the same input on three different hardware backends. Quality scores within 0.1. Docs capture every install step and every quirk hit.

**PR:** docs/sprint-day-2-kokoro-cross-hardware
**Time budget:** 6-10 hours (ROCm massaging is the wildcard)
**Done when:** Kokoro renders on AMD GPU + Apple Silicon + CPU + ROCm provider is verified, docs are detailed enough that the next contributor can replicate.

### Day 3 (Wed) — whisper.cpp on every backend + STT use cases wired

**Goal:** whisper.cpp transcribes Haystack News audio on every applicable backend, and STT is wired into the pipeline for transcripts, WER, round-trip scoring, and SRT.

**Tasks:**
- Build whisper.cpp from source on Hinoki with HIP backend: `make GGML_HIPBLAS=1`
  - Or: try the Vulkan path (`GGML_VULKAN=1`) as alternative — sometimes more reliable on RDNA4 than ROCm-via-HIP
- Build on Shiro with Metal/CoreML: `make GGML_METAL=1` (default on macOS)
- Build on Linux/CPU: vanilla `make`
- Download Whisper models: `bash models/download-ggml-model.sh base.en` and `medium.en` (large is overkill for v0.1.0)
- Write `src/stt.py` with five jobs:
  1. `transcribe(audio_path) -> str` — produces `transcript.txt` next to the MP3
  2. `transcribe_with_timing(audio_path) -> list[Segment]` — produces `transcript.srt` for accessibility
  3. `wer(audio_path, expected_script) -> float` — round-trip intelligibility metric
  4. `round_trip_score(audio_path, script) -> RoundTripReport` — per-segment WER, identifies which voices/registers are most/least intelligible per hardware backend
  5. `transcribe_for_corpus(audio_path) -> str` — emits a corpus-clean transcript for later voice training (queued for long plan)
- Run all five on a Haystack News test segment, on Hinoki ROCm + Shiro Metal + CPU
- Document backend-specific quirks in `docs/hardware/{amd-rocm,apple-silicon,cpu}.md`

**Exit criteria:** whisper.cpp transcribes the same input on three backends. All five STT use cases work. Round-trip scoring produces a comparison table showing WER per backend (should be near-identical if the audio is the same).

**PR:** feat/sprint-day-3-whisper-cpp-stt
**Time budget:** 5-8 hours
**Done when:** STT works on three backends and is wired into the pipeline.

**Note on integration:** whisper.cpp is a binary, not a Python package. `src/stt.py` shells out via `subprocess.run([...], shell=False)`. Keep it boring — no fancy bindings.

## Why this works as an MVP

1. **It picks the most hardware-portable model pair available.** Kokoro ONNX (provider abstraction) + whisper.cpp (compile-flag abstraction) + Stable Audio Open (PyTorch device abstraction) cover three different hardware-backend patterns in one repo. A contributor learning the OSS stack learns three different cross-hardware idioms.
2. **It prioritizes commercial deployability** by avoiding CC-BY-NC weights. Stable Audio Open replaces MusicGen for this exact reason. An operator running their own internet radio station can do so commercially without legal exposure.
3. **It uses Hinoki's RX 9070 as the AMD dev environment.** AMD-first is a real commitment to hardware diversity in OSS AI tooling, not an afterthought. ROCm massaging gets documented as part of the artifact.
4. **It produces a real public artifact** — a launchable v0.1.0-mvp tag with a working README, three hardware paths verified — that can be pointed at by AINW members, OSS aggregators, the AMD collaboration conversation, and anyone considering autonomous radio as a use case.
5. **It de-risks the long plan** by validating on real hardware before the engine abstraction refactor. Phase 2 of the long plan starts with knowledge of what actually behaves how on each backend.
6. **It establishes the bifurcation cleanly.** OSS and proprietary are visibly separate from day one. Future work knows which side it belongs on.

## Bottom line

`agent-radio-oss` is a new public repo under `ainorthwest/`. It's the engine-agnostic, license-clean, hardware-agnostic, commercially-deployable distribution of agent-radio. The MVP sprint creates it, ports a curated subset, integrates Kokoro ONNX + whisper.cpp + Stable Audio Open across AMD ROCm + Apple Silicon + CPU, and ships a v0.1.0-mvp tag: a working Haystack News episode rendered autonomously on AMD hardware, with transcripts, with quality scoring, with music beds, ready for a stranger to clone and run their own commercial AI radio station.

The hardware diversity isn't a workaround — it's the entire point. AMD, Apple Silicon, CPU now; NVIDIA in the long plan. Three different hardware-backend abstractions taught through three different models. The sprint succeeds when an operator with any of those hardware setups can stand up their own station.
